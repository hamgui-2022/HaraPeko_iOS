LEE JAEHYEOK（イ・ジェヒョク）

### アプリ名
**腹ペコフェンリル（HaraPeko）**

#### コンセプト
**「近くの獲物（お店）を、すぐ仕留める。」**
現在地周辺の飲食店を、狼（フェンリル）の狩りをモチーフにした世界観ですばやく探せるアプリ。

#### こだわったポイント
- API キーを xcconfig + .gitignore で分離し、ソース・コミットに含めない構成
- 位置情報の状態（取得中／取得完了／許可拒否）を出し分け、拒否時は設定アプリへ導線
- API が距離を返さないため、現在地と店舗座標から距離を自前で算出して表示
- 通信のローディング／エラー、ページングの状態を明確に管理
- 利用ガイドライン準拠のクレジット表記（Powered by / 画像提供）を全画面に実装

## アプリケーション機能

### 機能一覧
- 現在地取得：GPS（CoreLocation）で現在地を取得
- 逆ジオコーディング：座標から地名を表示（CoreLocation / CLGeocoder）
- レストラン検索：現在地・検索半径・ジャンル・予算でグルメサーチ API を呼び出し
- ページング：「もっと見る」で検索結果を追加読み込み
- レストラン情報表示：検索結果一覧・店舗詳細を表示
- 地図アプリ連携：店舗の所在地を Apple マップで開く
- 共有：店舗ページを共有（ShareLink）
- 予約導線：ホットペッパー店舗ページをブラウザで開く

### 画面一覧
- 検索条件入力画面：現在地・検索半径・ジャンル・予算を指定して検索する
- 検索結果画面：検索結果の飲食店を一覧表示する（ページング対応）
- 店舗詳細画面：選択した店舗の詳細（写真・住所・営業時間など）と地図を表示する

### 使用しているAPI, SDK, ライブラリなど
- ホットペッパー グルメサーチ API
- 標準フレームワークのみ：SwiftUI / CoreLocation / MapKit / Observation / Foundation(URLSession)
- 外部ライブラリ：不使用

### 利用技術（フレームワークや AI など）の選定理由
- SwiftUI：課題の原則に従い採用。宣言的UIで状態と画面の対応が明確。
- @Observable（Observation）：最新の状態管理。状態変化を View に自動反映できる。
- async/await + URLSession：標準のみで非同期通信を簡潔に実装し、外部依存を回避。
- MapKit：店舗のミニ地図表示に使用。
- CLGeocoder：逆ジオコーディング（座標→地名）に使用。preferredLocale で地名を日本語に固定。iOS 26 では非推奨だが、iOS 18 対応のため採用。
- 外部ライブラリ不使用：本規模では標準（AsyncImage 等）で要件を満たせるため。
- AI 利用について：実装・学習の補助として生成AI（Claude）を利用。生成コードは内容を
  理解した上で採用し、設計判断は自身で行った。

### デザイン面でこだわったポイント
- 「狩り」を想起させるダークテーマのデザインシステム（ember を主アクセント）
- カラートークン・角丸・余白を一元管理（Color 拡張）
- 書体に Pretendard JP を採用し、仮名・漢字のカバレッジと統一感を確保
- ネイティブの操作感（固定CTA・固定ヘッダー・Dynamic Type 対応）

### 既知の不具合・制限
- 電話アプリ連携は未実装：グルメサーチ API のレスポンスに電話番号が含まれない（tel はリクエスト用パラメータのみ）ため。代替として予約ページ導線を用意。
- お気に入り保存は MVP 範囲外のため未実装。

### 技術面でアドバイスして欲しいポイント
- Swift Concurrency（@MainActor / Task）の使い方が適切か
- ViewModel と View の責務分担・画面間のデータ受け渡し設計のスケーラビリティ
- 画像読み込み・キャッシュ戦略の最適化（標準 AsyncImage の是非）
- 動作対象OSを iOS 18.0 に設定したが、対応端末カバレッジと最新API活用のバランスとして妥当か
- （開発フローについて）GitHub の Issue / Pull Request は、実務ではどのような粒度・運用で使われているか
